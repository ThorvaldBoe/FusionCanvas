# PRD Audit - 2026-06-17

## Purpose

This audit reviews the FusionCanvas PRD set against three questions:

- Is anything missing for FusionCanvas to become a working basic tool?
- Are there inconsistencies that need to be cleared up?
- Are there obvious improvements that would make FusionCanvas stronger?

The review covered the core documents, roadmap, phase PRDs, and individual feature PRDs currently in `Living Documents/PRD`.

---

# Executive Summary

The PRD set is directionally strong. The product concept is coherent: a local-first desktop workspace that organizes Print on Demand work around stores, niches, topics, listings, assets, and the Idea -> Concept -> Design -> Listing workflow.

The biggest risk is not lack of vision. It is MVP boundary creep and a few implicit dependencies. The current docs describe a useful first product, but the roadmap MVP list was missing some supporting Phase 2 records needed by the Basic Concept, Design, and Listing tools. The strategic decisions now clarify that the MVP includes a minimal AI slice for idea generation and concept improvement, with image generation and other advanced AI work deferred.

The corrections made alongside this audit:

- Added `FC-0011 - Stage Tool Host` to the Phase 0 scope and acceptance criteria.
- Expanded the roadmap MVP boundary to include the support PRDs required by the Basic Concept, Design, and Listing tools.
- Clarified that the MVP includes a minimal `FC-0401` and `FC-0402` slice.
- Strengthened the data model MVP recommendation to include Concept, Design, Mockup, and Mockup Product Settings records.
- Clarified the distinction between `WorkflowStage` and lifecycle `Status`, with status limited to Draft, Published, Paused, and Rejected.
- Updated asset ownership expectations so imported files are copied into a managed FusionCanvas workspace.
- Added `Core Documents/Strategic Decisions.md` for fundamental choices.

# 1. Missing for Basic Functionality

## Mostly Covered

The PRDs already cover the core working tool:

- application shell
- local SQLite persistence
- workspace file storage
- store, niche, group, and listing management
- navigation tree
- tags, search, filtering
- listing inspector
- stage navigator
- tabbed document window
- context-aware tools
- stage tool host
- idea capture
- concept versions and design triangle
- design records and asset relationships
- prompt history
- basic listing metadata
- mockup records and manually configured mockup product settings

That is enough to define a basic but genuinely useful local-first creative workspace.

## Missing or Previously Implicit

### MVP support records

The roadmap MVP boundary included the Basic Concept Tool, Basic Design Tool, and Basic Listing Tool, but did not explicitly include several supporting PRDs those tools depend on:

- Concept Versions
- Design Records
- Asset Relationships
- Manual Mockup Records
- Listing Metadata Editor
- Design Triangle View
- Creative History Timeline

These have now been added to the MVP boundary.

### Minimal AI dependency slice

The MVP boundary now includes AI-assisted idea generation and concept improvement. That requires:

- provider abstraction
- AI settings
- provider capability checks
- user-visible privacy boundaries
- graceful fallback when no provider is configured

The roadmap now states that `FC-0401` and `FC-0402` are part of the MVP, limited to the provider and settings support needed for idea generation and concept improvement.

### Asset ownership behavior

The product now has a clear asset ownership decision: imported files are copied into a managed FusionCanvas workspace. The database stores paths to managed workspace files instead of depending on original external locations.

### Basic workspace safety

The basic tool would benefit from future PRDs or explicit sections for:

- workspace backup/export
- database schema migration policy
- missing-file recovery
- sample/demo workspace data
- first-run workspace creation

These do not block the first product design, but they matter before real users depend on the tool.

# 2. Inconsistencies and Corrections

## Phase 0 omitted Stage Tool Host

The roadmap includes `FC-0011 - Stage Tool Host`, and later PRDs depend on it. The Phase 0 document listed context-aware tools but did not list the Stage Tool Host in scope.

Correction made: Phase 0 now includes `FC-0011` in scope and acceptance criteria.

## MVP boundary depended on unlisted Phase 2 features

The Basic Concept, Design, and Listing tools depend on concept, design, mockup, metadata, history, and asset relationship records.

Correction made: roadmap MVP boundary now includes those support PRDs.

## Workflow stage and lifecycle status were overlapping

The docs used Idea, Concept, Design, and Listing both as stages and statuses. That is convenient language but risky implementation.

Correction made: `Data Model.md` and `FC-0105` now recommend separating `WorkflowStage` from operational `Status`.

Decided direction:

- `WorkflowStage`: Idea, Concept, Design, Listing, Archive
- `Status`: Draft, Published, Paused, Rejected

## Data model initial recommendation was too small for the stated MVP

The previous data model MVP recommendation omitted Concept and Design, even though the MVP boundary includes Basic Concept and Design tools.

Correction made: the data model now recommends Concept, Design, Mockup, Mockup Product, Mockup Template, and Mockup Color Variant records for the basic working MVP.

## Encoding artifacts exist in some documents

Some rendered tree diagrams and punctuation appear as `â...` artifacts in terminal output. The actual Obsidian rendering may still be acceptable, but this should be cleaned when convenient because it can confuse implementation readers and future automated processing.

No broad encoding cleanup was performed in this pass to avoid accidental document churn.

# 3. Obvious Improvements

## Keep MVP AI narrow

The MVP includes AI because it is central to the product, but the AI slice should stay narrow:

1. Generate ideas.
2. Improve concept triangle elements.
3. Preserve prompt history and context.
4. Keep manual editing available.

Generative image creation, listing text generation, critique/scoring, and broader automation should wait until the core workflow is stable.

## Treat stage tools as the main extensibility test

The Stage Tool Host is the bridge between MVP usability and the future plugin platform. Built-in Idea, Concept, Design, and Listing tools should follow the same contracts future plugins will use where practical.

This gives FusionCanvas extensibility without waiting for a full plugin marketplace.

## Build readiness checks early

The Basic Listing Tool's readiness checklist is a strong idea. Similar lightweight readiness should exist across stages:

- Idea has enough content to become a concept.
- Concept has idea plus phrase or graphic direction.
- Design has at least one usable variant.
- Listing has final design, mockups, title, description, price, and provider preparation data.

These checks should guide the user without blocking creative work.

## Keep marketplace integrations out of the basic listing tool

The docs are consistent that direct Printify and Shopify creation belongs in later integration specs or plugins. This is the right boundary.

The basic tool should prepare local source-of-truth data and exportable assets. Publishing should come later.

## Use strategic decisions before adding complexity

Several choices will shape the whole application:

- AI in MVP or manual-first MVP
- workflow stage vs lifecycle status
- separate domain tables vs generic navigation node table
- copied assets vs referenced assets
- official paid plugins vs community integrations

These are now captured in `Core Documents/Strategic Decisions.md`.

## ADDED Requirements

### Requirement: Mockup entity is part of the core domain model
FusionCanvas SHALL define a Mockup as a Phase 2 core domain entity that belongs to exactly one listing and preserves one product preview image record for that listing's Listing stage.

#### Scenario: Contributor reviews the core model
- **WHEN** a contributor inspects the domain model after this change is accepted
- **THEN** Mockup is represented as a named domain concept owned by a listing
- **AND** the model still does not require future concepts such as Marketplace Product, Performance Record, Plugin Data, or Workflow Template

#### Scenario: Mockup is item-bound
- **WHEN** a mockup is represented in the domain model
- **THEN** it belongs to exactly one listing
- **AND** no free-floating mockup exists outside an item
- **AND** the mockup is scoped to the Listing stage of that item

### Requirement: Mockup records preserve preview images for a listing
FusionCanvas SHALL allow a listing to have zero, one, or many mockup records, SHALL preserve superseded and rejected mockups without deletion, and SHALL allow each mockup to record product type, vendor product, template, color variant, view, notes, intended marketplace use, an optional showcased-design reference, and a source flag.

#### Scenario: User manually attaches a mockup image
- **WHEN** the user attaches an existing image as a mockup for a listing
- **THEN** FusionCanvas creates a new mockup record with the manual source flag
- **AND** links the image asset to the mockup through asset-management
- **AND** records any supplied product, template, color, view, and marketplace metadata

#### Scenario: User creates a mockup for a specific design
- **WHEN** the user creates a mockup that showcases a particular design of the listing
- **THEN** FusionCanvas references that design in the mockup's showcased-design field
- **AND** rejects a cross-listing design reference

#### Scenario: Superseded mockups remain available
- **WHEN** a mockup is superseded by a newer one
- **THEN** FusionCanvas marks the prior mockup superseded
- **AND** preserves the record, its asset references, and its metadata
- **AND** keeps it reachable for later review without cluttering the active mockup gallery

### Requirement: Mockups reference assets through asset-management
FusionCanvas SHALL associate a mockup with one or more asset files through the existing asset-management context-link model, with Mockup as a valid asset context kind, and SHALL NOT delete referenced assets when a mockup is removed.

#### Scenario: User links an asset to a mockup
- **WHEN** the user links an asset to a mockup
- **THEN** FusionCanvas creates the asset context link through asset-management
- **AND** the asset remains owned by its store and reachable from its other contexts

#### Scenario: Mockup removal preserves referenced assets
- **WHEN** a mockup is removed
- **THEN** FusionCanvas removes the mockup and its asset context links
- **AND** preserves the underlying asset records unchanged

#### Scenario: Asset removal preserves the mockup record
- **WHEN** an asset linked to a mockup is removed through asset-management
- **THEN** FusionCanvas removes the asset context link from the mockup
- **AND** preserves the mockup record, metadata, and remaining asset references

### Requirement: Manual and generated mockups share one entity with a source flag
FusionCanvas SHALL store a source flag on each mockup record identifying whether the mockup was manually attached or generated by the Basic Listing Tool, and SHALL store a regeneration-metadata block on generated mockups sufficient to regenerate the mockup when the source design or template changes.

#### Scenario: Manual mockup is created
- **WHEN** the user manually attaches a mockup image
- **THEN** FusionCanvas sets the source flag to manual
- **AND** does not require a regeneration-metadata block

#### Scenario: Generated mockup is created
- **WHEN** FC-0212's Basic Listing Tool generation flow creates a mockup
- **THEN** FusionCanvas sets the source flag to generated
- **AND** stores the regeneration-metadata block with source design, mockup product, template, color variant, placement parameters, and generation timestamp

#### Scenario: Generated mockup can be regenerated
- **WHEN** the user requests regeneration of a generated mockup whose source design or template has changed
- **THEN** FC-0212's generation flow uses the regeneration-metadata block to produce a new mockup
- **AND** FC-0206 marks the prior mockup superseded without deleting it

### Requirement: Mockup metadata is documented and preserved
FusionCanvas SHALL store mockup product type, vendor product, template, color variant, view, notes, and intended marketplace use as documented mockup metadata, and SHALL preserve unknown metadata keys during edits.

#### Scenario: User edits mockup metadata
- **WHEN** the user edits the product, template, color, view, or marketplace metadata of a mockup
- **THEN** FusionCanvas persists the changes atomically
- **AND** preserves the mockup's identity, asset references, showcased-design reference, and source flag

#### Scenario: Unknown metadata is preserved
- **WHEN** a mockup carries metadata keys outside the documented mockup set
- **THEN** mockup edits preserve those unknown keys unchanged

### Requirement: Mockup operations persist atomically
FusionCanvas SHALL persist mockup create, edit, asset link, supersede, reject, and removal operations as atomic snapshot operations in the active workspace database, reloading the latest snapshot before each mutation and saving once per operation.

#### Scenario: Workspace reloads after mockup operations
- **WHEN** a successful mockup operation is followed by an application or workspace database reload
- **THEN** the resulting mockups, metadata, asset references, lifecycle markers, and source flags match the last successful persisted state

#### Scenario: Persistence fails during a mockup operation
- **WHEN** the repository cannot save a mockup operation after an optimistic projection
- **THEN** FusionCanvas reports a recoverable error
- **AND** restores the last confirmed mockup, asset link, and metadata state
- **AND** retains recoverable user input needed to retry when applicable

### Requirement: Mockups are dependent records for listing and design deletion
FusionCanvas SHALL treat mockup records as dependent creative records for the listing and design permanent-deletion guards, SHALL block permanent deletion of a listing or design that has one or more mockups, and SHALL require explicit mockup cleanup before a connected record can be deleted.

#### Scenario: Listing with mockups cannot be permanently deleted
- **WHEN** the user requests permanent deletion of a listing that has one or more mockups
- **THEN** FusionCanvas blocks the deletion
- **AND** explains that the mockups must be removed or the listing archived instead

#### Scenario: Design with mockups cannot be permanently deleted
- **WHEN** the user requests permanent deletion of a design that has one or more mockups showcasing it
- **THEN** FusionCanvas blocks the deletion
- **AND** explains that the mockups must be removed or the design archived instead

#### Scenario: Listing or design without mockups remains deletable
- **WHEN** the user requests permanent deletion of a listing or design with no mockups or other dependent records
- **THEN** the existing deletion behavior applies without a mockup blocker
